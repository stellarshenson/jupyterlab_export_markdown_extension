Abstract

The Voluntary Milking System (VMS) Predictive Maintenance project aims to develop a machine learning-based system for identifying and preventing VMS malfunctions and breakdowns. Initially, the project attempted to predict breakdowns directly from VMS logs, but this was hindered by the lack of labeled data. The work evolved through three phases: unsupervised anomaly detection, alarm-based dataset construction, and time-to-failure regression experiments.

Anomaly Detection

An unsupervised autoencoder model detects anomalies in VMS operations by analyzing each milking event in isolation. The model outputs an anomaly score based on reconstruction error, which is aggregated across milking events and features to provide a daily health score for each VMS. The VMSDatasetBuilder utility enables construction of training datasets with configurable healthy/quarantine ratios around alarm events.

Data Engineering

High-frequency VMS logs (300 Hz, 150+ features) posed significant engineering challenges. The project developed transformation pipelines including duplicate removal, interpolation, Butterworth filtering, FFT-based frequency features, and event isolation. These techniques reduced noise while preserving predictive signal.

Alarm Proximity Regression

Once reliable alarm data became available, we attempted time-to-failure regression to predict when specific alarms would occur. Two approaches were developed: XGBoost with Pseudo-Huber loss and a TCN-Transformer deep learning architecture. Results showed that regression is viable for select alarm types (R² up to 0.50 for alarm 2_405), though many alarm types lack sufficient predictive signal. Key challenges include unknown causality between co-occurring alarms and lack of clear failure modality mapping.

Scalable Processing

The anomaly scoring system was deployed on AWS using batch processing with multiprocessing techniques, reducing processing time for 59,000+ logs from weeks to approximately one day.

Current Status and Future Directions

The anomaly scoring model provides a foundation for predictive maintenance. Further progress requires high-quality labeled field data to refine anomaly thresholds and validate regression models. Recommendations include focusing regression efforts on predictable alarm types and pursuing failure modality grouping with Product Development.

This document outlines the data science and engineering perspective on building the VMS Predictive Maintenance system.

Project Overview

The goal of this project is to develop a predictive maintenance solution for the Voluntary Milking System (VMS) at DeLaval. Initially, the focus was on predicting breakdowns, but due to the lack of labeled data, the approach shifted to an anomaly detection system. This document outlines the steps taken, the current status of the project, and future directions.

Project Link: VMSPRED

Current Status

We have developed and implemented an anomaly scoring system and anomaly scoring model, which has been performing scoring across over 59,000 logs from various VMS units. This system generates scores based on different features within the logs, and while the scores have been collected, further analysis and conclusions are still in progress.

Previously, our approach centered on building an anomaly detection model that output a pointwise-based score. This method evaluated how anomalous a specific feature was at a given point in time (t) within the VMS logs. However, this pointwise scoring approach did not lead to the results we had hoped for.

Our current scoring system takes a more holistic approach. Instead of evaluating features at individual points in time, it calculates an anomaly score for each milking event, which is then aggregated into a daily score. This has provided a broader perspective on the system’s health and behavior over time.

Before this shift in approach, we were attempting to use the pointwise anomaly scores to classify breakdown events using field data. Unfortunately, due to the lack of labeled data, we were unable to classify anomalies with enough certainty to create a reliable breakdown classifier. As a result, we were unable to develop a breakdown forecasting model that could predict system failures with a high degree of accuracy.

In addition to this we have developed data processing pipelines and anomaly scoring analysis solution

Approach Overview

Our approach to achieving predictive maintenance for the VMS (Voluntary Milking System) involves a multi-step process:

  1. Unsupervised Learning: The first step is to develop an unsupervised learning model to identify anomalies in the VMS logs. This approach detects unusual patterns across different features in the logs, flagging them as potentially anomalous behaviors.

  2. Supervised Learning: The next step builds on the identified anomalies by using a supervised learning model. This model classifies anomalies into two categories: harmless anomalies and those that could lead to various malfunctions in the VMS system. This stage depends on obtaining labeled data from the field to train the classifier effectively.

  3. Regression for Breakdown Prediction: The final step involves using a regression-based approach. The goal here is to map the internal states of the system into a probability of breakdown, predicting not just if a breakdown will occur, but also when. This involves estimating the remaining useful life (RUL) of the VMS unit before failure.

This three-step approach is central to our strategy for developing a robust VMS predictive maintenance system.

Previous Attempt: Pointwise Anomaly Detection

Our earlier approach involved developing a model that analyzed VMS logs using a sliding window. Within each window, the model calculated error metrics to detect anomalies, employing an unsupervised learning technique. The objective was to assess the differences between consecutive log entries and identify potential anomalies in the VMS system.

However, this approach faced several challenges:

  1. VMS Operational Variance: The sliding window model attempted to comprehend varying operational states of the VMS, such as milking, cleaning, and idle periods. This introduced a high variance in the data, making it difficult for the model to consistently detect anomalies across different types of VMS behavior.

  2. Difficulty in Interpreting Anomalies: The pointwise anomaly scores generated by the model were difficult to interpret. Identifying meaningful anomalies in the context of normal system operations proved challenging, leading to inconclusive results.

  3. Lack of Labeled Data: The absence of sufficient labeled data from the field made it impossible to effectively compare detected pointwise anomalies with real-world breakdowns or malfunctions. This hindered our ability to validate the model's effectiveness in predicting meaningful system failures.

Due to these limitations, we decided to pivot to a more holistic daily anomaly scoring system, which provides a more comprehensive overview of VMS behavior and allows for more accurate anomaly detection at the day level rather than pointwise.

Current Approach: Milking Event-Based Anomaly Scoring

The current approach focuses on individual milking events rather than analyzing the entire duration of VMS logs. This shift brings several benefits:

  1. Higher Data Consistency: Milking events exhibit more uniform behavior across different VMS units worldwide. By focusing on these similar events, we reduce the entropy and variance in the data, which enables a clearer understanding of the system's normal and anomalous behaviors.

  2. Independent Event Evaluation: Each milking event is treated as an independent sequence, allowing us to assess anomalies on an event-by-event basis. We use a fixed window size (T) to encompass the duration of the milking event. If the event is shorter than T, the window is padded with null data. If the event exceeds T, the event is discarded - however, discarding a small fraction of events (0.5% to 1.5%) does not harm the overall predictive power. These longer-duration events can be analyzed separately in future iterations once the prediction quality on standard events improves sufficiently.

  3. Improved Model Training: This approach enables better sample shuffling and tuning for the unsupervised learning model. The model evaluates these independent sequences, which leads to more precise anomaly detection for individual milking events.

  4. Anomaly Aggregation: Although the model produces pointwise anomaly scores within the time window T, these scores are aggregated to generate a single anomaly score per milking event. Further aggregation across all milking events in a day results in an anomaly score per day per VMS. This final daily anomaly score serves as the primary output of the system.

By focusing on individual milking events, this method provides more structured data for the unsupervised learning algorithm, leading to a more effective and interpretable anomaly scoring system.

Processing Anomaly Scores

One challenge we encountered during anomaly analysis was the unbounded nature of anomaly scores, making them difficult to compare and benchmark across different models. To address this, we implemented a process to aggregate and bound the anomaly scores effectively. Here are the key steps we took:

  1. Aggregating Anomaly Scores Across Milkings: Since the number of milking events varies across different VMS units and days, summing anomaly scores wouldn’t yield meaningful results. We explored three approaches to aggregate the daily anomaly score:

    • Simple Mean: Takes the average of all anomaly scores from the day’s milking events, providing a balanced representation.
    • Harmonic Mean: Promotes lower anomaly scores, minimizing the impact of high values.
    • Geometric Mean (chosen approach): For scores greater than 1, geometric mean amplifies outliers relative to the rest of the distribution, effectively capturing the significance of anomalies across the day's milking events.
    • Aggregating Feature-Based Anomaly Scores: After calculating the anomaly score per feature across all milking events, we used a geometric mean again to combine the scores into a single anomaly score per VMS per day. This approach helps preserve high anomaly scores, signaling that something unusual is happening in the VMS.

  2. Bounding Anomaly Scores: The final step was to bound the anomaly scores to make them comparable. We applied a hyperbolic tangent function with a scaling factor, gamma, to bound the scores between 0 and 1. This bounding approach was adopted specifically because we were unable to calibrate proper anomaly thresholds due to the lack of labeled data - the tanh function ensures scores remain bounded and comparable regardless of the raw score magnitude.

This process of aggregation and bounding allows us to achieve consistent, comparable anomaly scores that effectively highlight significant anomalies in the VMS operation.

In Development - Weighted Feature Contribution to Anomaly Scores

Currently, we aggregate anomaly scores across different features using a geometric mean. However, this approach may not persist as it assumes that all features are equally significant in determining anomalies. In reality, not all features contribute to the final anomaly score in the same way. Some features may appear normal but still lower the overall anomaly score, even if they should be considered insignificant.

Note: In later implementations, we adopted the CancelOut mechanism (Borisov et al., 2019) which provides learnable weight factors that help understand feature contribution and dampen those that bring less value. This is implemented in the autoencoder and regressor models.

To address this, I am working on a weighted feature contribution approach. The idea is to assign different weights to each feature based on its importance in detecting anomalies. This would ensure that the most relevant features have a higher impact on the final anomaly score, while less significant features do not disproportionately influence it.

Bayesian Inference with Metropolis Algorithm

The approach will leverage Bayesian inference, specifically using the Metropolis algorithm (a form of Markov Chain Monte Carlo - see Metropolis Algorithm - Statistics How To). Here's how it will work:

  1. Initial Weights Assignment: Each feature will start with an initial weight that influences the overall anomaly score for each VMS.

  2. Expert Feedback Loop: A team of semi-experts will review the daily anomaly scores generated for various VMSs. They will evaluate whether the anomaly score accurately reflects any issues with the VMS's performance. Based on their assessment, they will provide feedback, essentially "thumbs up" (anomaly indicates a real issue) or "thumbs down" (the score is not indicative of a real problem).

  3. Bayesian Adjustment: This expert feedback will feed into the Metropolis algorithm, which will gradually adjust the weights of the individual features. Through multiple iterations, the algorithm will learn which features have the most influence on accurately predicting anomalies.

  4. Final Distribution: Over time, this process will converge on a final distribution of feature weights. The weighted contributions will reflect the actual behavior of the VMS, improving the accuracy and reliability of the anomaly scores.

By using this Bayesian inference approach, we will create a more sophisticated anomaly detection system that is adaptable and continually refined through feedback. This method ensures that each feature's impact on the anomaly score is dynamically and accurately adjusted based on its significance in identifying real issues within the VMS.

Solution Performance

The anomaly scoring system, developed to analyze the entire dataset representing 542 tracked VMSs, has only recently been implemented. However, the detection and scoring of cleaning events have not yet been incorporated. As a result, we have not been able to fully evaluate the accuracy of the anomaly scoring model at this stage.

During preliminary analysis, we observed that VMSs missing from the training data showed inflated anomaly scores. This suggests that certain aspects of their behavior were not captured in the training dataset, leading to higher anomaly detection in these systems. This finding motivated the creation of a new dataset that includes all VMSs to ensure more representative and reliable anomaly detection. To validate the model's ability to detect actual anomalies, we specifically tested models on known anomalous data - reconstruction error consistently spiked on these samples. While this empirical testing is limited, it was the only validation approach available without comprehensive labeled data.

The upcoming performance evaluation will require engagement from the entire team and access to labeled ground truth data from the field. In the absence of complete field data, one proxy measure we aim to integrate into the pipeline is the percentage of completed milkings, which could provide an indicator of VMS performance that correlates with anomalies detected by the system. This enhancement will be critical for further refining the model and scoring capabilities.

Engineering effort
Noteworthy is that the motivation driving the development of a much larger dataset—one that encompasses all of the VMSs represented equally across the entire spectrum—required major engineering efforts. This undertaking involved significant time and resource investment to arrive at a solution capable of handling such a large volume of data. The goal was to ensure that even with this massive dataset, the system would still perform efficiently, allowing for training to be executed within a reasonable time frame. Additionally, the solution had to be designed to avoid system crashes and the need for excessive computational resources, which would be cost-prohibitive. This balancing act between performance, scalability, and resource feasibility was a critical aspect of the engineering process.

Processing of 1000d of vms data requires at peak 130 GB of RAM and up to 14 GB (after optimisations) of GPU VRAM to perform the training and processing efficiently.

Conclusion of the approach

The VMS Predictive Maintenance System follows a structured approach aimed at preventing machine breakdowns in voluntary milking systems (VMS):

  • Data Preparation: Initial focus on cleaning and aligning VMS log data for analysis.
  • Unsupervised Learning Model: Detects anomalies in machine features and generates daily anomaly scores to flag potential issues.
  • Feedback Tuning: Incorporates feedback from the field (e.g., thumbs up/down) to adjust the weight of individual feature anomalies, improving the anomaly score's accuracy.
  • Supervised Classifier: Requires high-quality labeled data from the field to train a classifier that categorizes anomalies as malfunctions or breakdowns.
  • Regression Model: Further development of a model to predict breakdown likelihood, remaining useful life, and overall machine health.
  • Traffic Light System: Provides intuitive health indicators (green, yellow, red) on DeLaval Plus for farmers to monitor VMS status.

A critical next step is gathering labeled data from the field to improve the supervised classifier, ensuring more accurate anomaly identification and system effectiveness. This will help build a scalable and actionable predictive maintenance solution.

Data and Dataset Preparation

A significant portion of the VMS Predictive Maintenance project workload has been dedicated to data engineering and feature engineering due to the complexity of working with high-frequency, sparse VMS logs. Each VMS log contains millions of rows sampled at around 300 Hz (averaged value across multiple datasets with varying sensor frequencies) across more than 150 features, many of which are sparse. This presents a unique challenge in processing and selecting the most informative features for the anomaly detection model.

Original data is available on S3 (location will change): s3://vms-predictive-maintenance/data/interim/stage1_logs_processed

Key Challenges in Data Preparation

One of the primary tasks for a data scientist in this project was to carefully manage the balance between signal and noise when selecting features.

  • Too many features: Increases model complexity and introduces noise, particularly when sample size is limited relative to feature count. For autoencoders, irrelevant features risk the model learning to reconstruct noise patterns rather than meaningful signal, degrading anomaly detection sensitivity. While modern deep learning shows overparameterized networks can generalize well with proper regularization, careful feature selection remains important for interpretability and computational efficiency.
  • Too few features: Selecting too few features can result in underfitting, where the model oversimplifies the problem and loses predictive power by failing to capture important operational characteristics of the VMS.

This delicate balance required the implementation of several feature selection techniques to ensure that only the most relevant features were passed into the model.

Data Exploration

As part of our data exploration phase, our team applied several techniques to understand the data, assess feature significance, and observe potential clustering behavior before applying any machine learning algorithms.

We began by using dimensionality reduction methods, such as PCA (Principal Component Analysis), to observe potential data clusters. Although we noticed some clusters, they were largely inconclusive. We tried labeling data and correlating it with events like breakdowns, aiming to uncover patterns that could help with future predictions. However, this approach did not provide clear insights or direct solutions.

Other techniques we explored included non-linear dimensionality reduction methods such as t-SNE and UMAP. These methods aimed to reveal hidden structures by preserving local data relationships. Unfortunately, even with these advanced techniques, we could not distinguish between breakdowns and normal operations based on cluster behavior.

While these initial data exploration efforts didn’t yield actionable results, they were essential in ruling out simpler statistical or machine learning approaches (e.g., decision trees, random forests, SVMs). Ultimately, this guided us toward the more sophisticated anomaly detection and learning models that we implemented later on.

Feature Selection Techniques

We employed a combination of statistical and algorithmic techniques to evaluate feature importance and select the optimal set of features. Some of the key methods include:

  • Correlation Analysis: Evaluating the relationship between features to understand which ones are strongly correlated with the target variables.
  • Mutual Information: This non-linear metric helped us measure how much information one feature contains about another, particularly the target variable.
  • Recursive Feature Elimination (RFE): This method allowed us to iteratively remove less important features based on model performance until we arrived at a subset that significantly contributes to the model’s accuracy.
  • Principal Component Analysis (PCA): This dimensionality reduction technique was used to identify the most informative features. PCA helped simplify the dataset by creating a loading matrix that highlights the features with the greatest variance, and thus the most information, from the dataset's perspective.

Final Feature Set

After applying the above techniques, we reduced the original list of 150 features to approximately seven key features, which were found to be most significant for anomaly detection. These features capture the essential aspects of the VMS operation while reducing noise and complexity.

By focusing on these key features, we were able to improve both the efficiency and accuracy of the anomaly detection model, ensuring that the dataset used in training was both representative and manageable.

Note: Extensive testing of different dimensionality configurations was performed throughout the project. In later experiments with regression models, the feature set was expanded to the 30-50 range to capture additional predictive signals.

The final feature set used by the model are:

  • S_MainVacuum: main vacuum level
  • S_MainVacuum_FFT: dynamics of the vacuum level changes
  • S_HydraulicOilPressure: oil pressure in the hydraulic system
  • S_HydraulicOilPressure_FFT: dynamics of the oil pressure changes
  • S_ReceiverLevel: receiver tank level for milk
  • S_ReceiverLevel_FFT: dynamics of the receiver tank level for milk
  • S_MilkYield: mean value of milk yield between 4 milk yield lines
  • S_MilkYield_FFT: dynamics of the milk yield lines
  • S_MilkYield_Spread: difference of milk yield between lines
  • S_VacuumCup: mean value of vacuum cup pressure
  • S_VacuumCup_FFT: dynamics of the changes of pressure in vacuum cups
  • S_VacuumCup_Spread: difference between pressures in 4 vacuum cups
  • CupInGripper: cup in gripper signal (signifies operation of the gripper)
  • S_HydraulicOilTemp: hydraulic oil temperature
  • SYN_LBL_STATE_Milking_LAG: how much time passed since last milking
  • SYN_LBL_STATE_Milking_DUR: measure of the current milking duration

Dataset Processing and Feature Engineering

In the VMS Predictive Maintenance Project, processing the raw VMS logs was a complex yet essential task due to the high-frequency sampling (300 Hz) and the sparsity of the data. Each log contains millions of rows, with a significant portion of the data being non-informative. This led to the need for a well-defined preprocessing approach to transform the raw data into a usable format for modeling. Here’s a summary of the key steps taken during dataset preparation:

1. Removing Duplicates

A large number of duplicate entries existed in the raw data, and removing them was the first step toward simplifying the dataset. Eliminating these duplicates helped reduce noise and made the dataset cleaner for further analysis.

2. Removing Unnecessary Columns

We consulted with subject matter experts to identify columns in the dataset that were non-informative. Removing these columns allowed us to reduce the dimensionality of the data and improve model efficiency without sacrificing valuable information.

3. Data Imputation with Linear Interpolation

To fill in missing values, we used linear interpolation. This method was chosen because it provides a smooth estimate of the missing data points based on nearby values. Given the high performance of the Polars library, which is multi-threaded and optimized for speed, linear interpolation was computationally efficient and did not distort the underlying trends in the data.

Missing data in VMS logs is primarily attributed to lower reporting frequency of certain sensors rather than sensor failures. Given that some signal variability exists between readings, linear interpolation provides a reasonable approximation. Future work includes exploring PCHIP interpolation and explicit missing value handling using sentinel values that neural networks can interpret as missing data.

4. Resampling for Uniform Sampling

Machine learning models, particularly deep learning algorithms, struggle with unevenly sampled data. Resampling ensures that the data is uniformly spaced in time, improving the model’s ability to interpret patterns and relationships. This step was essential for converting the data into tensors for deep learning.

5. Noise Reduction with Rolling Averages

To reduce the variance caused by noise in the data, we applied a rolling average. This technique smooths out short-term fluctuations while preserving the overall trend. Thanks to the speed of the Polars library, rolling averages were computed rapidly, even on datasets with millions of rows.

Window size was treated as a hyperparameter and optimized using Bayesian optimization. Due to the lack of field labels at the time of research, expert opinion guided initial window selection based on visual inspection of the resulting signals. In later iterations, Butterworth filters replaced rolling averages for noise removal - these low-pass filters preserve low-frequency signal components while removing high-frequency noise. Butterworth filters also proved effective for smoothing step-like signals (such as anomaly spikes) to ensure neural networks obtain meaningful gradients and avoid local minima.

6. Feature Engineering

a. Power Spectrum Distribution (PSD)

We utilized a Power Spectrum Distribution (PSD) approach based on Welch’s algorithm, which normalizes the frequency distribution of a feature. The PSD provides a meaningful representation of the dynamic behavior of features over time and allowed us to reduce the sampling frequency without losing key information. The result was a more compact dataset, faster training, and a model capable of capturing long-term patterns in the data.

The value featuring PSD is in fact a mean of the weighted frequencies in the PSD for the window at position $t+\frac{T}{2}$ where $T$ is the window size. Window sizes ranging from 1 second to 1 minute were tested as hyperparameters, matching the final sampling frequency to achieve appropriate frequency resolution for the FFT analysis.

b. Flattening of Features

Certain features, such as milk yield or vacuum pressure, are measured multiple times per milking event (e.g., one measurement per teat). Instead of treating each measurement as a separate feature, we used flattening, which reduced the dimensionality by calculating both the mean and variance of these measurements. This process reduced the feature set, preserving the most important information without overloading the model with redundant data.

c. Event Isolation

Event isolation is a critical technique in our pre-processing pipeline, particularly for isolating individual milking events. Each event is padded to ensure that all events have a uniform duration. This makes it easier to create a consistent dataset, where events are treated as independent samples that can be shuffled and analyzed as needed. This method ensures that our data is ready for machine learning and deep learning frameworks that require consistent input sizes.

Example: Anomaly Scoring Pipeline

The anomaly scoring system for VMS predictive maintenance uses a streamlined data processing pipeline with scikit-learn pipelines. This ensures efficient preparation of the dataset for the anomaly scoring model. The steps in the pipeline are as follows:

  1. Select Input Features:

    • Start by selecting the essential features from the dataset, reducing its size to optimize processing speed.

    • Label States and Alarms:

    • Apply one-hot encoding to states such as milking and cleaning to easily identify and select milkings as events.

    • Remove Duplicates:

    • Eliminate any duplicate records from the dataset to ensure accuracy.

    • Interpolation:

    • Perform interpolation over numerical values to handle data sparsity and fill in missing values effectively.

    • Resampling:

    • Resample the data to 1-second intervals to ensure uniformity before applying further processing steps.

    • Flattening (Vacuum Cup Features):

    • Reduce the number of vacuum cup features from 4 to 2 by measuring variance and calculating the mean, preserving key information.

    • Flattening (Milk Yield):

    • Similarly, apply flattening to milk yield features, reducing the feature count from 4 to 2.

    • PSD (Power Spectral Density) Calculation:

    • Perform PSD calculation using Fast Fourier Transform (FFT) over most features, creating a new feature that captures the dynamic behavior of the original feature in the frequency domain.

    • Rolling Averages:

    • Apply rolling averages over the primary features and their FFT derivatives to smooth out data and reduce noise.

    • Milk Duration:

    • Generate a feature that measures the duration of each milking event, allowing us to detect anomalies related to unusually long or short milkings.

    • Time Since Last Milking:

    • Create a feature to capture how much time has passed since the last milking, providing important context for the current event.

    • Further Resampling:

    • Downsample the log to a lower frequency (e.g., 4 or 10 seconds), as the dynamics of features have already been captured in earlier steps.

    • Memory Optimization:

    • Reduce memory usage by lowering the precision of certain data points without sacrificing accuracy, reducing dataset size by around half.

    • Drop Unnecessary Columns:

    • Drop the state column and unnecessary timestamps as they are no longer needed after isolating milking events.

    • Milking State Isolation:

    • Perform event isolation, placing individual milking events one after another, padding where necessary to standardize the duration of each event.

    • Select Features for the Model:

    • Select only the required features for the model, ensuring consistency in the feature set for all models.

    • Column Sorting:

    • Sort columns into a specific order, ensuring the data is structured consistently for the model.

    • Scaling:

    • Finally, apply standard scaling to normalize the features, preparing the dataset for anomaly detection. The scaler is fit on the training set only and then applied to validation and test sets to prevent data leakage.

Implementation and Deployment

This pipeline is implemented as a scikit-learn pipeline, which ensures that all data processing steps are performed in sequence. Once the pipeline is created and fitted to the dataset during training, it is saved using JobLib as a pipeline model. This pipeline model is deployed alongside the trained anomaly detection model.

During inference, when new data is passed to the model, the pipeline first preprocesses the data in exactly the same way it was done during training. This ensures consistency and correctness in how the data is prepared, making sure the model receives properly preprocessed data at all times.

By deploying the pipeline together with the model, we maintain identical data transformations, ensuring that there is no discrepancy between how the training data and inference data are handled.

VMSDatasetBuilder

The VMSDatasetBuilder class is a core utility in the VMS Predictive Maintenance library (lib_vms_predictive_maintenance) that generates structured datasets from VMS logs. It processes raw telemetry data into analysis-ready datasets with configurable mixtures of healthy operational periods and quarantined periods near alarm events.

Key capabilities:
- Configurable healthy/quarantine ratio for class balance control
- Date, time, and VMS filtering via TimeslotFilter configuration
- Scikit-learn compatible transformation pipelines (single-log and master)
- Parallel processing with configurable batch size and workers
- Automatic metadata generation (JSON specification file)
- S3-based data sources with local caching for efficiency

Pipeline Flow

The dataset building process follows an 8-step pipeline:

Raw S3 Logs → Alarm Retrieval → Timeslot Identification → Sampling →
Single-Log Transform → Master Transform → Farm Enrichment → Output Dataset
  1. Generate unique dataset ID
  2. Retrieve alarms for quarantine determination
  3. Identify healthy and quarantined timeslots
  4. Sample timeslots to reach target dataset size
  5. Apply single-log transformation pipeline (per file)
  6. Apply master transformation pipeline (concatenated dataset)
  7. Enrich with farm metadata
  8. Write dataset and specification files

Healthy vs Quarantine Classification

Data is classified based on proximity to alarm events:

  • Healthy (quarantine=0): Periods with no alarms within the buffer window
  • Quarantined (quarantine=1): Periods within min_days_before_breakdown before or min_days_after_breakdown after an alarm event

The alarm_types parameter specifies which alarm codes trigger quarantine periods, using the major_minor format (e.g., "2_402"). Different combinations of standoff values were tested as experimental parameters to determine adequate quarantine periods around breakdown events.

Basic Usage

from lib_vms_predictive_maintenance.batch.vms_dataset_builder import VMSDatasetBuilder

builder = VMSDatasetBuilder(
    description="Production dataset for anomaly detection",
    min_days_before_breakdown=5,
    min_days_after_breakdown=3,
    dataset_length_days=180,
    dataset_healthy_ratio=0.8,
    dataset_sampling_interval_s=60,
    alarm_types=["2_402", "1_315"],
)

builder.info()
dataset_df = builder.build()
summary_df = builder.summary(dataset_df, plot=True)

Alarm Proximity Regression

Motivation

Once reliable alarm data became available and we developed sufficient understanding of different alarm types within the VMS system, we attempted regression-based time-to-failure prediction. As outlined in Step 3 of the Approach Overview, the goal was to map sensor telemetry patterns to a probability of breakdown, predicting not just if a breakdown will occur but when.

This proved to be a challenging undertaking. Alarms tend to co-occur temporally and there exists an unknown causality structure between them - multiple alarm codes may fire simultaneously or in rapid succession, making it difficult to isolate root causes. Our objective was to identify failure modalities of the VMS (hydraulics failure, vacuum failure, motor failure, gripper failure, etc.) that could serve as labeled events for supervised learning.

Alarm Selection Challenges

The VMS system generates over 500 distinct alarm types. Mapping these to meaningful failure categories proved highly fluid - we experimented with different alarm collections, attempting to identify which codes genuinely indicated impending mechanical failure versus transient operational issues. Product Development experts often provided contradictory guidance on how to interpret and include specific alarms. An alarm might be classified as critical by one expert and ignorable by another, or the same failure mode might manifest through different alarm codes depending on VMS configuration or firmware version.

We proceeded with a pragmatic selection of alarms showing consistent occurrence patterns, reasonable lead time between sensor anomalies and alarm triggering, and expert consensus where achievable.

Target Variable Design

Rather than treating alarms as categorical classification targets, we framed the problem as time-to-failure regression - predicting when a specific alarm event will occur based on sensor telemetry patterns. The regression target represents temporal proximity to alarm events:
- Value of 1.0 at the alarm event itself (T=0)
- Value decreasing toward 0.0 as distance from alarm increases
- Computed over a configurable lookback window (typically 3-7 days before alarm)

Initial approach: Linear proximity using VPM_LinearProximityTransformer - a simple ramp from 0 to 1 over the time window T-x to T, where T is alarm time and x is the lookback period.

Refined approach: The linear target creates abrupt gradient changes at window boundaries, which neural networks struggle to learn smoothly. We applied Butterworth low-pass filtering to the proximity curve, producing a more natural gradient that improves neural network training stability and convergence.

Model Architectures

Two parallel approaches were developed:

XGBoost Regressor - Ensemble of decision trees trained sequentially using Pseudo-Huber loss for robustness to outliers. Hyperparameters optimized via Optuna with 4-fold cross-validation. Fast to train and interpret, this approach outperformed deep learning models.

TCN-Transformer Deep Regressor - Neural network combining temporal convolutions with self-attention:
- CancelOut - Learnable gates that automatically identify which input features matter most
- TCN - Dilated convolutions (1, 2, 4, 8, 16) capturing patterns at multiple time scales while respecting temporal causality
- Multi-Head Attention - Learning which time points are most relevant to the prediction

Conclusions from Regression Work

What Worked
- Time-to-failure regression is viable for select alarm types with R² up to 0.50
- XGBoost with Pseudo-Huber loss provides robust baseline performance
- Butterworth-smoothed proximity targets improve neural network training
- Datasets focused on alarm-adjacent periods (0% healthy ratio) outperform balanced datasets

What Did Not Work
- Many alarm types show insufficient predictive signal (negative R²)
- Deep learning models underperform gradient boosting despite architectural complexity
- Tweedie loss function degraded performance across all experiments
- Some alarms co-occur in ways that confound individual prediction

Limitations
- Lack of clear failure modality mapping limits interpretability
- Expert disagreement on alarm significance complicates target selection
- Unknown causality structure between co-occurring alarms

Recommendations
- Focus resources on predictable alarm types (2_405, 2_402) rather than universal coverage
- Pursue failure modality grouping with Product Development to create semantically meaningful targets
- Consider multi-task learning to exploit correlations between related alarms

Model Development Overview

The model development for the VMS Predictive Maintenance Project is divided into three key stages:

  1. Unsupervised Learning Models
    The initial focus was on unsupervised learning to identify anomalies in the VMS data. This involved developing models that could process sequences of log data and highlight anomalous behavior without the need for labeled data. The model would analyze the data, detect anomalies, and output a reconstruction error, which would serve as the anomaly score.

  2. Anomaly Classification Models
    The second phase aimed to classify the detected anomalies as either harmless or indicative of potential malfunctions or breakdowns. This classification was based on labeled data, where the goal was to determine whether anomalies were benign or associated with system breakdowns.

  3. Regression-based Models
    The final stage involved developing a regression-based model to predict the probability of a breakdown and estimate the remaining useful life of the VMS. This model would map an internal state of the machine to breakdown probabilities and timelines, providing actionable insights for predictive maintenance.

Unsupervised Learning Model Evolution

Autoencoder Explanation:
An autoencoder is a type of neural network designed to learn efficient representations of data. It works by compressing the input data into a lower-dimensional representation (encoder), and then reconstructing the original data (decoder). The difference between the input and output (reconstruction error) serves as a signal for detecting anomalies.

To ensure the autoencoder learns from healthy data only, we developed a sophisticated dataset builder that allows for configurable cleaning of data around alarm events. This approach excludes anomalous periods from training data, ensuring the model learns to reconstruct normal VMS behavior.

LSTM Based Model
The unsupervised learning journey began with an LSTM-based autoencoder, a sequence processing model, to capture anomalies in time series data. The LSTM architecture, however, struggled with vanishing and exploding gradient problems, leading to inefficient learning and suboptimal anomaly detection. This method didn’t provide the desired accuracy due to its limitations in handling long sequences effectively.

LSTM Transformer Model
Next, the architecture evolved into a hybrid LSTM-Transformer model. The addition of transformers improved the model's ability to capture long-range dependencies across the sequence. Transformers use attention mechanisms, allowing the model to consider relationships across the entire input sequence simultaneously. This enhancement proved effective, but LSTM still struggled with its inherent limitations.

Conv1D FCN Model with Attention
The architecture then shifted to a fully convolutional network (FCN) with attention. THis model shown significant improvement in performance vs LSTM model. Shorter training time and better sequence comprehension.

This model required the Sequence Encoding using Time2Vec encoding (see [1907.05321] Time2Vec: Learning a Vector Representation of Time) to learn sequence representation as trend, seasonality and sequence.

This is currently the model deployed to Dev and one that produces scoring across 59k+days of over 500+ VMS operation

In Development: TCN With Attention
Experiments began with a Temporal Convolutional Network (TCN - see Temporal Convolutional Networks and Forecasting - Unit8). TCN, with its dilated convolutions, was better suited to handle time series data, especially when combined with attention mechanisms, which had already proven valuable in previous iterations.

Optimizing the Model

Several techniques were employed to improve model performance, including:

  • Skip Connections: These residual connections helped mitigate issues with vanishing gradients, making the network deeper without loss of information.
  • Positional Encoding (Time2Vec): This allowed the model to better comprehend the temporal aspect of the data, with Time2Vec encoding cycles and sequences.
  • Hyperparameter Tuning: Initially, hyperparameters were tuned using a toy dataset to avoid the time and computational cost of training on the full dataset. Once the model architecture was stable, further tuning was planned using techniques such as grid search or Bayesian optimization (Optuna - A hyperparameter optimization framework) on cloud infrastructure.

The models were trained on a toy dataset first to refine the architecture, and then validated on the actual VMS logs. Currently, the TCN-based model with attention and positional encoding is being tested.

This progression from LSTM-based architectures to TCN with attention highlights the importance of continually evolving the model to adapt to the complexity of the data and the specific requirements of the anomaly detection task.

Training Data Set and Model Training

Initially, the technique for producing the training data set allowed for building relatively small datasets, typically covering around 40 to 60 VMS logs over a few days. This size was already sufficient to saturate the model and lead to lengthy training times. Additionally, the model had a high number of parameters, which further extended the training time. Due to the limitations of processing, smaller time windows were selected initially, as the model struggled to handle larger time windows effectively and quickly ran into suboptimal results.

The data set construction technique evolved significantly over time. At first, data sets were built using manual transformations via a set of transformation techniques that had been developed. These techniques were applied both during the development of training data as well as inference. Eventually, this process became more streamlined through the development of a pipeline of transformers, using the SKLearn pipeline model. The pipeline approach helped to automate and standardize the transformations, allowing for more efficient model deployment and training.

The data set building technique itself also evolved from manually selecting VMS logs and days to developing a more advanced dynamic approach. This new technique adjusted the stand-off time for breakdown candidates and selected the least anomalous data from the cloud. Data were divided into time slots, and anomalous time slots were rolled out, allowing for a more balanced representation of VMS in the final dataset. This approach ensured that VMS were equally represented across the dataset, and configurable parameters controlled the duration and number of days selected for each VMS (see: Data and Dataset Preparation).

Note: The approach of using model output to select "least anomalous" training data was later abandoned due to concerns about circular logic - this risked training on the "easiest" examples and lacked interpretability. The current approach uses fixed standoff days with alarm indicators as proxies for anomalous periods.

To further improve dataset processing, a data specification (spec) in JSON format was introduced. This enabled verification of which VMS were represented in the initial training data set and ensured transparency. During the training process, it became clear that VMS not present in the training data set often appeared anomalous in the model's scoring system, likely due to their unique settings and behavior not represented during training. This insight helped refine the selection process for training data.

As of now, the data set consists of 1084 days of data, representing 542 VMS, with each VMS contributing roughly two days of data to the final dataset. This refined dataset construction process has been recently deployed.

Train/Validation/Test Split: A holdout set is always used for model verification. To prevent data leakage, we enforce VMS-day grouping - all milking events from a single VMS-day stay entirely within one split (train, validation, or test). This ensures that no information from the same day's milkings leaks across splits, maintaining the integrity of model evaluation.

Developing this data building capability required a significant amount of time due to memory constraints and the need to streamline the building process through multiprocessing.

Handling such large volumes of data (around 1-2 GB per day of data from VMS) required a robust and efficient approach. This process was optimized through multi-processing techniques, and the development of the data processing pipelines successfully addressed memory leaks and other challenges. The final data builder is now a fully configurable class that can be easily adapted for both data and processing pipelines.

Dealing with Overfitting

The author has employed several techniques to address overfitting in the models. Initially, when LSTM was used as the primary architecture, the only effective method to manage overfitting was through the application of regularization and dropout. The regularization used was L2-based, focusing on activation regulation.

As the models evolved and became deeper, overfitting was further reduced by incorporating identity skip connections (also known as residual connections) and batch normalization. When CNN-based models were introduced alongside attention mechanisms, batch normalization was replaced with layer normalization. Although batch normalization would still be effective due to the large batch sizes (ranging from 256 to 512, and now up to 2048), layer normalization has been recommended by several recent AI publications and was considered more appropriate. Batch size was treated as a hyperparameter and tested across this range, balancing GPU memory constraints, compute capabilities, convergence quality, and gradient noise trade-offs.

Additionally, the author applied a regularization technique involving the addition of Gaussian noise during training. This noise was randomly introduced at the start of each training batch, which helped prevent the model from overfitting to specific noise patterns. The decision to remove original noise from the dataset was made to avoid the model learning static noise patterns, whereas the Gaussian noise added during training changed with each batch, making it harder for the model to fit to.

This approach has proven to be effective, though overfitting can still appear explosively after running models for between 1,000 to 2,000 epochs over large datasets. The extended training duration (2,000-5,000 epochs with 1,000-epoch early stopping patience) is intentional - strong regularization terms slow convergence but improve generalization. Convergence verification was limited by the lack of labeled data; expert opinion guided model selection based on training curves and empirical testing on known anomalous samples.

Loss and Error Metrics

Currently, for the autoencoder, we primarily use Mean Squared Error (MSE) as the core metric to evaluate reconstruction performance. However, we also track two additional metrics to provide a broader view of the model's behavior:

  • Cosine Similarity: This metric is used to track the shape divergence between the output and the input, ensuring that even if the magnitude changes, the overall shape and trend of the time series remain consistent.
  • Mean Absolute Error (MAE): MAE is monitored to keep an eye on the absolute error, providing insights into the average deviation between predicted and actual values.

In addition to these, we are exploring other potential metrics, such as:

  • Mutual Information: This entropy-based metric measures the shared information between input and output signals. It is a strong candidate for integration due to its differentiability, making it suitable for use in the model’s learning process.

  • Dynamic Time Warping (DTW) and FastDTW: These metrics measure the similarity between time series with temporal distortions, providing an alternative way to evaluate how well the autoencoder captures sequential behavior. Although DTW is not fully differentiable, it remains valuable for scoring purposes rather than as a direct loss function.

  • Cross-Entropy: This metric can serve as a proxy for KL Divergence, as both are related to comparing two probability distributions. In the context of the autoencoder, Cross-Entropy helps evaluate how well the reconstructed outputs approximate the original inputs by measuring the difference between their respective probability distributions. Minimizing Cross-Entropy implicitly minimizes the divergence between the input and output distributions, making it useful for assessing probabilistic aspects of reconstruction quality.

  • Hybrid Metrics: We are considering combinations of multiple metrics using harmonic means and logarithmic aggregations. For example, combining MSE, MAE, and Cosine Similarity through harmonic means can balance their influence. Furthermore, the use of non-linear transformations like tanh can smooth metric combinations and help manage outliers or large errors in a more robust manner.

The combination of these metrics is still under development, with mutual information and hybrid approaches being particularly promising due to their potential to provide a holistic view of the model's performance while maintaining differentiability and robustness.

Hyperparameter Tuning

Hyperparameter tuning was an essential step in optimizing the performance of the predictive maintenance models. Initially, the models were tuned using a toy dataset to allow for faster iterations. These toy datasets were synthesized with anomalies placed at specific points in the time series. This approach allowed the model training to proceed quickly, which in turn facilitated hyperparameter tuning across a grid of parameters within a reasonable timeframe.

The author made an effort to reduce the complexity of hyperparameter tuning by focusing on a subset of key parameters. To further simplify, the Optuna framework was employed. Optuna, a hyperparameter optimization framework based on Bayesian statistics, was used to run experiments and find the optimal set of hyperparameters.

Once the optimal parameters were identified through Optuna, the author proceeded to manually fine-tune certain parameters by running additional experiments on the actual dataset. The results of the most notable experiments, along with their corresponding parameters and findings, were saved in a local experiments repository. Additional Training Engineering

Several techniques were implemented to improve the training process, many of which were added as callbacks:

  • Early Stop Mechanism: The author implemented an early stopping mechanism with a threshold of 1,000 epochs. The total length of training typically ranged from 2,000 to 5,000 epochs. This was necessary due to the heavy regularization applied to stabilize the model's convergence, which slowed down the training process.

  • Learning Rate Decay: Another important technique was learning rate decay. Starting from a rate of 0.01, the decay factor was 0.5, with a patience setting between 100 and 500 epochs. This allowed the model to search for parameter optima using large steps and gradually slow down to smaller steps when no further progress was made.

  • Custom Stop Callback: A custom callback was developed that allowed the author to stop training manually by removing a training lock file, ensuring the training could be halted without interrupting the final evaluation.

  • Documentation: The model training notebook automatically generated markdown files, which documented the parameters used, any modifications made, and additional notes from the author regarding the data and parameter changes that influenced the model.

  • Training Monitoring: TensorBoard was used to monitor the training process, especially during hyperparameter tuning. For local training on the GPU, the author implemented a custom visual callback to track the progression of metrics across epochs, ensuring that training was progressing towards the optimal set of parameters.

These techniques helped ensure the model's efficient training and the continual improvement of the anomaly scoring system.

Anomaly Processing

Anomaly processing in the VMS Predictive Maintenance project had to be handled as a batch process. Initially, the processing was streamlined through multithreading and sequential execution. However, this approach proved to be highly resource-intensive and time-consuming. With approximately 50,000 days of logs to process, the entire batch took around 4-5 days, which was unacceptable.

To improve efficiency, we developed a multithreaded approach where computationally intense tasks were still serialized due to Python's Global Interpreter Lock (GIL). However, downloading logs and performing I/O operations were handled in parallel using multithreading. This new approach reduced processing time to about three days, though it still consumed significant CPU and memory resources. Additionally, we encountered issues with multithreaded processing in some libraries like Polars and TensorFlow, where locks would cause inconsistent behavior.

To further optimize, we shifted to a multiprocessing approach using the loky library, which offered better handling of serialized models and deployment across separate processes. This avoided many of the limitations caused by Python's standard multiprocessing library. Additionally, we implemented a shared memory cluster with a queueing mechanism, allowing processes to reuse shared resources effectively. This approach removed the need to reload models and pipelines for every process, further reducing resource consumption.

By employing this multiprocessing solution with loky, we were able to fully saturate 16 processors and 64GB of RAM in our cloud environment, deployed via Amazon Elastic Container Service (ECS). Processing time was reduced to roughly one day, a significant improvement over earlier methods. We also introduced dynamic scaling for the number of workers, depending on memory requirements, which ensured optimal resource usage while keeping cloud costs in check.

The processed logs are stored in S3 buckets in parquet format, ready for further analysis.

Configuration

The configuration for the anomaly processing is managed through a hot-swappable .ini file, which is stored in an S3 bucket. The ECS anomaly processing task pulls this .ini file, along with the resources it references, to execute the processing. The .ini file specifies the model, the processing pipeline, various operational parameters, the link to the original data source from which the data for processing is pulled, and the target directory where the anomaly scores developed during processing are stored.

Additionally, the anomaly processing is hot-swappable not only because of the .ini file but also because the VMS Predictive Maintenance library (lib_vms_predictive_maintenance) is dynamically pulled from S3. This means that every time the task executes, it uses the latest version of the library, ensuring that the system runs with the most up-to-date code and improvements without requiring manual redeployment of the task.

Conclusion and Next Steps

As of now, the anomaly scoring system has been developed, and the scoring model has matured enough to produce a reasonable anomaly score using well-defined metrics. Additionally, the engineering solution for batch processing of the anomaly score has been developed, optimized, and deployed in the development stage.

A critical part of the process after deployment is the ability to verify these anomaly scores against the actual operational systems. Gathering as much feedback as possible is essential to determine whether VMSs are experiencing trouble or showing signs of malfunction, and correlating this feedback with the corresponding anomaly scores. This will help in validating the system's accuracy and reliability.

The anomaly score produced by the system could be a highly reliable tracking solution due to the nature of how autoencoder models work. The only potential obstacle that could make the anomaly score inconclusive is if the training data is insufficient. Improvements in the quality and quantity of the datasets used for training the anomaly detection model could further enhance the scoring accuracy. As of now, the system has tracked and rounded up data from 542 VMSs, aggregating up to two days of operation taken from different days in the target dataset used for training. We aim to expand this dataset further, and this will also apply to cleaning events. Capturing anomaly scores from both milking and cleaning will provide a more holistic picture, leading to more conclusive results.

Next Steps

Our current efforts focus on refining the anomaly scoring system and developing new capabilities:

  1. Cleaning Anomaly Scoring: We plan to introduce a system that processes the anomaly scores for VMS cleaning events and allow for joint analysis of both milking and cleaning anomaly scores. The initial focus on milking events was deliberate - milking is the most complex state of the robot and should exhibit the strongest anomaly signals. Cleaning event integration is planned once milking-based detection achieves sufficient signal quality.
  2. Optimizing Feature Weights with Bayesian Inference: A key part of our future efforts will be using Bayesian inference with the Markov Chain Monte Carlo (MCMC) method, specifically the Metropolis algorithm, to optimize the weights assigned to different features in the anomaly scoring system.
    • To automate this process, we plan to incorporate the percentage of completed milkings per day for each VMS into the scoring system. This metric will help gauge the reliability and health of the VMS on a daily basis.
    • By collecting thumbs-up and thumbs-down feedback from the field (indicating whether the anomaly score reflects actual VMS behavior), we can feed this information into the Bayesian inference model. Over time, the system will automatically adjust the weights of individual features, ensuring that the final anomaly score better reflects the importance of each feature and provides a more meaningful assessment of VMS health.
  3. Field Testing: Once the system is fine-tuned, we will validate its accuracy in real-world conditions.
  4. Three-Tier Alert System for Farmers: We aim to introduce a "traffic light" system for farmers, offering green (everything is fine), yellow (potential issues), and red (likely breakdown) alerts on the DeLaval Plus system.
  5. Geographical Data Refinement: We will explore regionalization to account for differences in VMS configurations and behavior across different regions, ensuring that models are tailored to specific geographical requirements.
  6. Incorporating Alarm Criticality: Future iterations will include critical alarm information, helping to further refine the anomaly scores.

This enhanced system will provide a robust anomaly detection framework that can be deployed widely, giving farmers actionable insights into the performance of their VMS and enabling early detection of potential issues.
